### Introduction

LLM은 여전히 사전학습 이후에 새롭게 들어오는 많은 정보들을 통합하는 데에

어려움을 겪고 있음. 본 연구는 해마의 **long-term memory indexing theory**에

영감을 받아 새로운 정보를 더 깊고 효율적으로 통합하는 패러다임을 제시함

**Hippocampal Indexing Theory**

- **Pattern separation**
    
    새로운 경험이나 지식을 받아들일 때 기존의 기억들과 뒤섞이지 않게
    
    각각을 **고유하고 독립적인 기억으로 분리하여 저장함**
    
- **Pattern completion**
    
    **일부의 단서**만 가지고도 그날의 전체 기억을 온전히 끄집어내는**(검색)** 능력
    

⇒ **관계성 기반의 인덱스**를 만들어서 **짧은 질문만으로도 정확한 정보를 찾아내자**

### Methods

<p align="center"><img src="{{ '/assets/board/2026/06/HippoRAG/image_0.png' | prepend: site.baseurl }}"  max-width="100%" height="auto"></p>

HippoRAG는 Offline indexing과 Online retrieval 단계로 구분됨

**Offline Indexing (~Pattern Separation)**

1. LLM + OpenIE 방식으로 문서에서 Triple 리스트 추출
    
    `Doc -> LLM -> List[Triple]`
    
2. Lexical matching의 실패를 우회하기 위해 retrieval encoder로 각 노드를
    
    dense embedding으로 만듦
    
    `Node -> Encoder -> Embedding`
    
3. Dense embedding에서 의미가 비슷한 노드끼리는 인공 엣지를 추가함

이 단계에서는 세 가지가 반환됨

- **Indexed Passages:** 일반적인 inverted index, `KG Node → List[passage_idx]`
- **Knowledge Graph:** OpenIE로 추출된 명사구 노드들로 구축된 네트워크
    
    그리고 각 노드가 문서에 얼마나 등장했는지, P 행렬도 추가로 만들어짐
    
    $P \in \mathbb{R}^{|N| \times |P|}$
    
- **Dense Index:** KG node의 dense embedding

**Online Retrieval (~Pattern Completion)**

1. query에서 LLM + NER로 entity list 추출 (query named entities)
    
    `query -> LLM -> List[entity]`
    
2. 각 entity별로 가장 가까운 node를 찾고, 이 query node부터 KG 탐색
3. PPR 알고리즘으로 KG를 탐색
4. Indexed passages에 대한 PPR node probs를 모으고 ranked list를 반환함
- **Node specificity**
    
    더 검색을 잘하기 위해 IR에서는 보통 IDF 지표를 사용함. 그러나 이것은
    
    global importance score이므로 뇌과학적으로는 말이 안 됨. 따라서 이에
    
    대한 대안으로 $s_i=|P_i|^{-1}$ 사용. 의미는 Node i가 전체 문서 중 몇 개에
    
    등장했는가. 이 점수를 PPR 시작 전 query node에 곱해줌

- $R_q$ nodes & neighbors에서 알 수 있듯이 PPR로 pattern completion하지 않고
    
    1-hop 이웃을 그대로 사용하는 것은 오히려 큰 성능 하락을 유발함
    
- Node specificity는 검색에 큰 도움이 됨
- Synonymy edges를 연결하는 것은 noisy entities 환경일수록 많은 도움이 됨

전반적으로 성능이 좋음. 그러나 검색 시에 매번 KG 내에서 PPR을 수행해야 하는데,

엣지 수에 비례해서 연산량이 늘어나므로 computation overhead가 클 것으로

예상됨. 또한 (H, R, T)의 triple에서 Relation에 대한 정보 없이 독립적으로 인덱싱한

것이 조금 아쉬움.
